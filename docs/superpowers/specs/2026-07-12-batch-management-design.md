# 批次管理设计

## 目标

为数据核对引入批次。每个批次是一份独立的 Excel 数据快照，达人和视频记录都直接归属一个批次。管理员管理批次、导入和导出；普通用户在自己有数据的批次之间切换。

## 数据模型

- `batches`：批次元数据，包含 `id`、`name`、`year`、`month`、`title`、`status`、`previous_batch_id`、`source_filename`、`created_at`、`imported_at`。
- 批次名称由 `year`、`month`、`title` 生成，格式为 `YYYY年MM月｜ 自定义标题`，完整名称唯一。
- `darens.batch_id`：达人记录直接归属批次；昵称只在同一批次内唯一，约束为 `(batch_id, nickname)`。
- `videos.batch_id`：视频记录直接归属批次；约束为 `(batch_id, daren_id, platform, work_id)`。
- `users` 保持全局，通过 `display_name = darens.nickname` 识别本人在所有批次中的数据。
- 现有数据迁移到 `2026年05月｜ 数据核对`，并设为当前批次。

## 批次生命周期

批次只有 `draft`、`current`（已发布）、`history` 三种状态。

1. 管理员创建批次后，批次为 `draft`；系统最多保留一个草稿批次。
2. 草稿批次可删除，也可重复上传 Excel。
3. 每次草稿导入都在事务内写入达人、视频和新用户；失败时回滚本次数据，草稿保留，前端显示错误。
4. 导入成功后草稿仍为 `draft`，管理员显式发布时才变为 `current`，原 `current` 变为 `history`。
5. 发布时记录 `previous_batch_id`；撤销当前批次会将它恢复为 `draft`，并把 `previous_batch_id` 恢复为 `current`。
6. `current` 和 `history` 批次不可删除、不可再次导入；历史批次只读。
7. 新草稿导入或发布准备期间，普通用户继续查看原当前批次。

## 查询与权限

- 管理员可查看并切换所有当前和历史批次；草稿只在批次管理中可见。
- 普通用户默认进入当前批次；如果当前批次没有同名达人，展示“本期暂无数据”。
- 普通用户的切换器只列出自己有数据的批次，且只能访问 `nickname = session.display_name` 的达人和视频记录。
- 达人确认状态存储在批次达人记录上，因此每个批次独立。
- 达人状态统计、视频异常统计和导出均按所选批次执行。

## 接口与界面

- 新增管理员批次接口：创建、列表、删除草稿、发布、撤销、获取当前批次。
- 导入接口要求草稿 `batchId`；成功后返回批次信息和导入结果。
- 达人、视频、导出接口都接受并校验 `batchId`；管理员默认当前批次，普通用户只能使用自己的可见批次。
- 管理员工作台增加批次管理入口和批次选择器；草稿创建后在其中上传 Excel。
- 达人列表、视频页和导出按钮使用当前选择的批次。
- 普通用户工作台和数据页提供仅限本人数据批次的选择器。

## 导入与重复规则

- 同一批次的一次 Excel 内，重复的“昵称 + 平台 + 作品 ID”覆盖该批次前一行数据。
- 已发布的批次不能再次上传；需要更正时创建新批次。
- 不同批次的同名达人和相同作品 ID 都保留为独立记录。
- Excel 中未出现的达人不属于新批次，但其账号和历史批次数据保留。

## 验证

- 数据库迁移保留现有数据，并创建指定的初始当前批次。
- 草稿导入失败时不写入达人、视频或用户；草稿仍可重试。
- 发布操作原子切换当前批次，撤销操作原子恢复上一个当前批次。
- 达人、视频、确认状态、异常统计、导出均不能跨批次泄漏数据。
- 普通用户只能看到自己的批次和数据；管理员能查看历史批次。
